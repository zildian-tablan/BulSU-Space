Deployment Manual & Runbook

Updated November 2025. This deployment guide is written for future BulSU Space developers and admins who will maintain the platform. Follow it when rebuilding the system, installing dependencies, or deploying updates.

Environment

Consistent setup for every developer

Code steps keep every workstation aligned and avoid “works on my machine” issues.

Delivery

Repeatable build and deploy steps

Simple npm scripts, Firebase CLI commands, and quick checks make each deployment easy to repeat.

Reliability

Pre-flight + rollback guidance

Pre-flight lists, backup reminders, and rollback notes keep every rollout calm even under pressure.

Quick Deployment Checklist

  1. Gather essentials: Confirm Node.js 18 LTS, npm 9+, Firebase CLI 13+, and copy environment files from ENV_VARIABLES_NEEDED.md. Log in via firebase login using the deployment account.
  2. Prep the workspace: Unzip the latest project package, open it in VS Code, run npm install at the root, then repeat inside functions/ so both front-end and backend scripts share clean dependencies.
  3. Validate locally: Launch npm start in both terminals, hit http://localhost:3000, and trigger key scripts (presence monitor, test-* utilities) to ensure credentials and emulators behave before touching production.
  4. Ship confidently: Execute npm run build, deploy with npx firebase deploy --only hosting (plus rules/functions if needed), then follow the smoke-test checklist to watch logs and user journeys for the first 15 minutes.

Section 01

Baseline Requirements

Confirm these before unzipping the project archive or running npm install.

Workstation & OS
  • Windows 10/11, macOS 13+, or Ubuntu 22.04 LTS fully patched.
  • At least 8 GB RAM (16 GB recommended) and 10 GB free storage for node modules + build artifacts.
  • Stable internet with access to npm registry, Firebase Hosting, and Google APIs.
Tooling
  • Node.js 18 LTS (use nvm or fnm to match package.json engines).
  • npm 9+ and Firebase CLI 13+ (npm install -g firebase-tools).
  • Optional helpers: Git, VS Code, PowerShell 5.1+/Terminal, Postman.
Credentials & Secrets
  • Firebase project owner/editor access to bulsu-space production project.
  • Service account JSON for local admin scripts (store inside .secrets/, never commit).
  • Environment variables from ENV_VARIABLES_NEEDED.md (FCM keys, signaling endpoints, analytics IDs).

Section 02

Project Checkout & Dependency Install

Use this updated six-step recipe whenever you onboard a new device or refresh the environment.

  1. Extract the latest BulSU-Space.zip package into a workspace you control (example: C:/BulSU-Space). Keep the root folder name unchanged.
  2. Open VS Code (or your preferred IDE) and load BulSU-Space/project. All remaining commands assume this directory.
  3. Run npm install from the project root terminal. Do not interrupt; first run can take several minutes while Tailwind/CRACO deps compile.
  4. Open a second terminal, execute cd functions, then npm install (first-time only). After install, run npm start to spin up emulated functions/watchers.
  5. Return to the primary terminal and run npm start. This launches the React dev server, applies CRACO overrides, and watches Tailwind styles.
  6. Verify both terminals stay green. If one crashes, stop both, clear node_modules, and repeat the install process before continuing feature work.

VerifyDevelopment services

  • Front-end served at http://localhost:3000.
  • Functions emulator (if enabled) logs incoming requests.
  • Presence monitor scripts (under scripts/) respond when invoked manually.

OptionalEmulator extras

  • Run firebase emulators:start --only firestore,auth for offline QA.
  • Seed mock data using scripts/test-*.js utilities.

Section 03

Deployment Workflow

Use this streamlined flow for both staging and production. Swap the Firebase project via --project when pushing outside production.

Phase 01 Prepare build

Run npm run lint (if configured) and optional unit tests. Clean previous artifacts with Remove-Item build -Recurse -Force so stale files cannot leak into the release.

Phase 02 Compile assets

Execute npm run build. Confirm build/index.html, asset-manifest.json, and service worker files regenerate without warnings.

Phase 03 Deploy hosting

Run npx firebase deploy --only hosting. Include --project production-id (or alias) if you are not already targeting production.

Phase 04 Post-release checks

Visit the live site, test sign-in, messages, and notifications. Monitor Firebase Hosting + Functions logs for anomalies during the first 15 minutes.

Need to ship rules or functions? Run firebase deploy --only functions,firestore:rules,storage immediately after hosting so all layers stay in sync with firestore.rules, storage.rules, and functions/src.

Section 04

Extended Operational Needs

Use these guardrails to keep BulSU Space reliable while multiple colleges share the deployment.

Environment Variables
  • Maintain .env.local (dev), .env.production (CI), and keep a vaulted copy for DR.
  • Refresh values using ENV_VARIABLES_NEEDED.md whenever Firebase rotates keys.
  • Mirror CDN-specific flags via netlify.toml or vercel.json if alternate hosting is used.
Security & Access
  • Review docs/SECRETS_MANAGEMENT.md after every release cycle.
  • Rotate Firebase service account keys at least once per semester and immediately after staff turnover.
  • Require MFA on deployment accounts and lock unused roles in IAM.
Backups & DR
  • Export Firestore rules via deploy-database-rules.ps1 before every production deploy.
  • Snapshot Realtime Database using database.rules.backup*.json scripts.
  • Archive build/ as builds/<date>-release.zip so rollbacks stay under ten minutes.

Section 05

Pre-deploy Checklist

Code & Config

  • Latest master merged, linted, and pushed.
  • firebase.json, firestore.rules, storage.rules double-checked for pending edits.
  • Manifest icons, FCM keys, and messaging service worker versions confirmed.

Data Integrity

  • Backup critical collections or export to GCS prior to schema tweaks.
  • Validate Firestore indexes via firestore.indexes.json and deploy if new composite indexes exist.
  • Confirm any migration scripts in scripts/ completed successfully.

People & Comms

  • Notify admins via STATUS_REPORT channel or Teams chat.
  • Post maintenance notice on the announcements feed 30 minutes in advance.
  • Assign an on-call owner for post-release monitoring and incident intake.

Rollback Plan

  • Store the prior build zip locally and in cloud storage.
  • Record firebase hosting:versions:list output (version ID + timestamp).
  • Document revert steps for Firestore rules or indexes changed in this release.

Section 06

Smoke Tests After Deploy

Functional

  • Sign-in + MFA flow for both student and admin QA accounts.
  • News feed shows new announcements without manual refresh.
  • Presence indicator updates within 30 seconds when the browser closes.
  • Push notifications arrive via FCM (confirm latest firebase-messaging-sw.js shipped).

Operational

  • Firebase Hosting dashboard shows the new release ID and status green.
  • Functions/Firestore logs stay free of uncaught exceptions for at least 10 minutes.
  • Google Analytics (or GA4) receives events from staged user journeys.
  • Support desk + admins notified that deployment finished.

Section 07

Troubleshooting & Recovery

npm install fails

Delete node_modules and package-lock.json, run npm cache clean --force, then reinstall. If corporate proxies block downloads, toggle to mobile hotspot temporarily.

Firebase deploy auth error

Run firebase login --reauth, set the project alias via firebase use production-id, and retry the deploy.

Build succeeds, UI blank

Check build/index.html for missing %PUBLIC_URL% vars. Regenerate using the latest .env entries.

Functions emulator stuck

Kill orphan ports with npx kill-port 5001 8080, clear firebase-debug.log, then restart npm start inside functions/.

Rollback needed

Run firebase hosting:rollback with the saved version ID, or redeploy the last known-good zip from builds/<date>.

Presence scripts fail

Verify service account credentials used by presence-function-monitor.js and confirm Realtime DB rules were not tightened during release.

Need more? Review docs/SECURITY_IMPLEMENTATION.md, docs/SECRETS_MANAGEMENT.md, and flow/POSTING_FLOW.md when troubleshooting feature-specific regressions.

Billing & Owner Turnover

If you are performing a turnover of the Firebase project billing account and project ownership, contact cout.johnpaul.torres@gmail.com to initiate the handover. Below are the recommended steps, information to include in the handoff, and console (web UI) steps the billing contact can follow to add the new database owner/admin and update the billing account.

What to include in your initial email

  • Project ID: the Firebase/GCP project ID (example: bulsu-space).
  • New admin email: the Google account to be added as Owner/Admin (example: new-admin@example.com).
  • Billing change request: whether to transfer the billing account or link a different billing account (include billing account ID if known).
  • Preferred transfer window: date/time for the change and expected downtime (if any).
  • Confirm backups: note that Firestore/Realtime Database backups have been taken and where they are stored (GCS path or zip archive).

Console (recommended) steps the billing contact should follow

  1. Open the Firebase Console → select the project → click Project settings → Users and permissions. Click Add member, enter the new admin's email, and assign the Owner role (or the minimum set of roles you prefer).
  2. Open the Google Cloud Console → IAM & Admin → IAM. Verify the new member appears with the expected role(s). Add roles like Owner or targeted roles (e.g., Billing Admin, Firestore/Realtime DB Admin) if needed.
  3. To change billing: open Billing → select the billing account → Account Management → Manage members. Add the new billing administrator or transfer account control as appropriate.
  4. Confirm the project is linked to the intended billing account: Billing → Billing page → Projects linked to this billing account.

Post-transfer checklist

  • Verify the new admin can access Firebase Console, view/change rules, and deploy functions.
  • Rotate any service account keys shared during the handover and update any local .secrets/ or CI vault entries.
  • Deploy test rules and a small test release to ensure hosting, functions, and database permissions work under the new owner.
  • Document the change in STATUS_REPORT.md with the transfer timestamp, new owner email, and billing account ID.
Security note: Never send service account JSON files over unencrypted email. Use a secure file share or a vault (GCP Secret Manager, Vault) and rotate keys immediately after the handover.
